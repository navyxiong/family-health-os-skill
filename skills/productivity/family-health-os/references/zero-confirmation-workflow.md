# 零确认录入工作流

## 背景

用户反复纠正记录流程中的确认行为：
- "以后录入病历不要让我确认，你直接录入"
- "每次录入完成，汇报该次录入的是什么"
- "为什么每次资料录入还是要我确认两次？"
- "不需要任何确认。1. 病历录入时直接静默执行，完成后只输出一行摘要。2. 所有命令/操作一旦收到立即执行，不询问确认、不等待批准、不提示"Command Approval Required"。禁止任何中间确认步骤。直接执行到位。"
- **"复盘刚才的任务，你一共让我授权确认了十几次，不允许再出现这样的情况，接收到指令，全部由你自己直接完成，在输出结果前不要找我"**（2026-06-14，最强烈一次）

## 核心规则

### 禁止的行为（用户会计数每次确认请求）

1. **逐条询问确认** — 不能在录入每张图片/每条数据后问"是否确认"
2. **中途暂停等待** — 不能在录入流程中间暂停等待用户回复
3. **完整摘要作为确认** — 不能在录入后输出完整档案摘要作为确认步骤
4. **逐条列出所有检查项** — 不能输出每个检查项的详细列表
5. **等待用户说"开始"** — 收到图片后立即开始，不等待指令
6. **任何"是否需要..."句式** — 包含"是否需要我..."、"请确认是否..."、"要我继续吗"
7. **重复用户已经给出的信息** — 用户已经提供过的（患者姓名、文件路径、数据）不要回显确认
8. **多选让用户做决策** — 除非用户主动要求决策，否则不要给出"请选择：a/b/c"

### 强制输出格式

录入完成后，仅输出一行关键信息：

```
已录入。<类型> <日期> | <医院> | <事件数>事件
```

正确示例：
- `已录入。血常规 2026-05-26 | 湘雅医院老年医学科 | 52事件`
- `已录入。CTPA 2026-06-05 | 湘潭市一医心血管内科 | 54事件`
- `已录入。超声心动图 2026-04-01 | 湘潭市中心医院 | 58事件`

错误示例：
- ❌ 输出完整档案摘要作为确认
- ❌ 询问"这些数据是否正确？"
- ❌ 逐条列出检查项明细
- ❌ 输出多行详细描述
- ❌ "需要我帮你做 X 吗？"（主动询问）
- ❌ "请确认是否继续..."（任何形式的确认）

## 适用任务范围

零确认规则**不限于病历录入**，覆盖所有任务类型：

| 任务类型 | 正确行为 | 错误行为 |
|----------|----------|----------|
| 查询 | 直接查询+输出结果 | "是否需要我查询..." |
| 生成（时间轴/摘要）| 直接生成+输出 | "请确认是否生成完整报告" |
| 数据修改 | 直接修改+汇报结果 | "已修改 X，请确认是否正确" |
| 文件操作 | 直接执行 | "请选择保存到哪个目录" |
| 错误处理 | 自动重试/降级/用替代方案 | "工具失败，是否尝试其他方法？" |
| 多次工具调用 | 全部完成后一次性输出 | 每步单独输出等待确认 |

## 完整流程

```
用户发送图片/文本
  ↓
vision_analyze() 提取文字
  ↓
execute_code() 解析为结构化字典
  ↓
get_patient_by_name() 确认患者存在
  ↓
tool_layer.add_exam() / add_hospitalization() 写入
  ↓
rebuild_timeline() 自动聚合
  ↓
输出一行摘要
```

**关键点**：
- 整个流程在同一个工具调用链中完成
- 不能在中途暂停等待用户指令
- 如果用户发送多张图片，每张图片按顺序录入原始数据，不逐张询问确认
- 最后一次性输出总结摘要（仅当用户明确请求时）

## 批量图片处理

用户连续发送多张图片时：

1. 第一张图片：完整流程录入
2. 后续图片：
   - 同一检查类型：追加到同一 exam 的 images[]
   - 不同检查类型：创建新的 exam 记录
   - 都不暂停等待确认
3. 每次录入后立即调用 rebuild_timeline()
4. 最后一次性输出总结摘要

## 图片路径处理

系统提示的 image_url 可能存在延迟或错误：

```python
# 检验步骤
import os
if not os.path.exists(image_url):
    # 扫描 image_cache/ 目录找最新文件
    import glob
    cache_dir = '/path/to/image_cache'
    files = sorted(glob.glob(f'{cache_dir}/*.jpg'), key=os.path.getmtime)
    if files:
        image_url = files[-1]
```

**处理原则**：
- 系统提示中的 "Here's what I can see" 是辅助参考，权威数据源是 vision_analyze 返回的完整文字
- 两者不一致时，以 vision_analyze 结果为准
- 不暂停询问用户

## 输出前自查清单

每次准备输出回复前，自查以下问题：

- [ ] 本次回复是否包含"是否需要..."、"请确认..."、"要我..."等确认句式？
- [ ] 本次回复是否在中间位置出现"等待用户回复"的逻辑？
- [ ] 本次回复是否给出了 a/b/c 多选让用户决策？（除非用户主动询问）
- [ ] 本次回复是否复述了用户已经提供过的信息？
- [ ] 本次回复是否把本应一次性完成的工作拆成多步让用户确认？

如有任一项为"是"，删除对应的确认请求，直接给出结果。

## 纠正记录

本次会话中用户的纠正点：
1. 次数：录入后输出完整档案摘要作为确认
2. 次数：每张图片录入后输出详细列表
3. 次数：在录入中途暂停等待用户确认
4. 日期错误：6月15日被误认为湘雅入院日期，实际应为6月13日
5. 出院错误：6月12日被误标为出院，实际是转院建议
6. 日期混淆：CTA影像片日期被混淆为2026-05-20
7. 通用规则升级（2026-06-14）：用户说"我让你确认了十几次"，要求规则扩展到所有任务，不仅是病历录入
